# 为什么形状生成需要约束求解？

## 1. 问题本质：形状是相互依赖的

### 1.1 核心矛盾

**模糊测试生成 vs 形状约束**：

```
传统模糊测试：独立生成每个输入
  - 整数：随机 [0, 100]
  - 字符串：随机长度
  - 数组：随机大小

形状生成：输入之间存在依赖关系
  - 矩阵乘法：A.shape[-1] == B.shape[-2]  ← 硬性约束
  - 广播：dim_i(A) == dim_i(B) 或 dim_i(A) == 1 或 dim_i(B) == 1  ← 软性约束
  - 拼接：非拼接轴维度必须相同  ← 硬性约束
```

**关键洞察**：形状不是独立随机变量，而是满足特定关系的**联合随机变量**。

### 1.2 具体案例：矩阵乘法

假设生成一个矩阵乘法节点：

```
输入A: [M, K]    输入B: [K, N]
        ↓                ↓
       matmul(A, B) = [M, N]
```

**约束**：
1. `A.ndim == 2` 且 `B.ndim == 2`（至少 2 维）
2. `A.shape[-1] == B.shape[-2]`（K 维匹配）
3. 批次维度（如果 > 2D）可广播

**随机生成的问题**：
```kotlin
// ❌ 错误做法：独立随机生成
val shapeA = randomShape()  // [3, 5]
val shapeB = randomShape()  // [7, 2]
// 结果：matmul 失败，因为 5 ≠ 7

// ✅ 正确做法：约束求解
// 变量：A.dims[0], A.dims[1], B.dims[0], B.dims[1]
// 约束：A.dims[1] == B.dims[0]
// 求解：A.dims[1] = B.dims[0] = 随机值（如 5）
// 结果：A = [3, 5], B = [5, 2] ✓
```

### 1.3 更复杂的案例：多节点图

考虑一个 3 节点的计算图：

```
输入X: [N, D]
   ↓
 matmul(X, W1)  ← 约束1：X.dims[1] == W1.dims[0]
   ↓
 relu
   ↓
 matmul(relu_out, W2)  ← 约束2：relu_out.dims[1] == W2.dims[0]
   ↓
输出
```

**变量**：
- `N`, `D`（输入形状）
- `W1.shape = [D, H1]`（权重的形状取决于输入）
- `H1`（隐藏层大小）
- `W2.shape = [H1, H2]`（取决于前一层的输出）

**约束链**：
```
约束1：W1.dims[0] == X.dims[1] == D
约束2：W2.dims[0] == relu_out.dims[1] == H1
```

**如果没有约束求解**：
- 随机生成 `W1.shape = [K1, K2]`，可能 `K1 ≠ D` → 失败
- 即使 `W1` 正确，`W2` 也可能不匹配 `H1` → 失败
- **成功率 ≈ 0%**（除非运气好）

**使用约束求解**：
1. 先选 `N=32, D=64, H1=128, H2=10`（随机值）
2. 推导：
   - `X.shape = [32, 64]`
   - `W1.shape = [64, 128]`（因为 W1.dims[0] == D）
   - `relu_out.shape = [32, 128]`（保持形状）
   - `W2.shape = [128, 10]`（因为 W2.dims[0] == H1）
3. **成功率 = 100%**

## 2. 为什么"事后修复"不够？

### 2.1 当前的错误方案

**错误思路**：先随机生成，失败后再修复

```kotlin
// 错误的"事后修复"逻辑
fun generateGraph(): UirGraph {
    // 1. 随机生成拓扑
    val graph = LogicGenerator().generate()
    
    // 2. 随机分配形状
    assignRandomShapes(graph)
    
    // 3. 检查并修复（马后炮）
    for (node in graph.nodes) {
        if (!checkShapeCompatibility(node)) {
            fixShape(node)  // 修改形状
        }
    }
    
    return graph
}
```

**问题1：修复是局部的，可能引入新冲突**

```
节点A: matmul(X, W1)  → 修复 W1 形状
节点B: add(W1, W2)    → W1 已被修复，W2 可能不匹配
节点C: matmul(W2, W3) → W2 可能已被节点B修复，W3 又不匹配
```

**修复链**：A → B → C → ... → Z（指数级复杂度）

**问题2：某些约束无法事后修复**

```
算子：concat([A, B, C], axis=0)
要求：A.shape[1:] == B.shape[1:] == C.shape[1:]

如果随机生成：
  A = [10, 20]
  B = [15, 30]  ← dim 1 不匹配（20 ≠ 30）
  C = [12, 25]  ← 都不匹配

"修复"意味着什么？
  - 修改 B.shape[1] = 20？但 B 已经被其他节点使用
  - 修改 A.shape[1] = 30？影响所有依赖 A 的节点
  - 修改 C.shape[1] = ? ← 无解，因为 20, 25, 30 都不同
```

**结论**：事后修复在复杂依赖图中不可行。

### 2.2 正确思路：约束求解

**核心思想**：生成时就满足约束，而不是事后修补

```
方法1：贪婪实例化（简化版约束求解）
  - 按拓扑序遍历节点
  - 为每个节点生成满足约束的形状
  - 后续节点基于前面已确定的形状
  
方法2：符号约束求解（完整版）
  - 所有形状先用符号表示（N, M, K...）
  - 收集所有约束
  - 一次性求解，找到满足所有约束的具体值
```

**对比**：

| 方法 | 复杂度 | 成功率 | 支持的约束类型 |
|------|--------|--------|----------------|
| 随机生成 + 事后修复 | O(N²) | < 50% | 仅简单约束 |
| 贪婪实例化 | O(N) | 90-95% | 大部分约束 |
| 符号约束求解 | O(N + E) | 100% | 所有约束 |

## 3. 广播：为什么需要约束求解？

### 3.1 广播规则的本质

**NumPy 广播规则**：

```
A = [a1, a2, a3]
B = [b1, b2]

广播步骤：
1. 右对齐：
   A = [a1, a2, a3]
   B = [  , b1, b2]

2. 逐维比较：
   dim 0: a1 vs ?  → OK（B 没有这一维）
   dim 1: a2 vs b1 → 必须满足 a2 == b1 或 a2 == 1 或 b1 == 1
   dim 2: a3 vs b2 → 必须满足 a3 == b2 或 a3 == 1 或 b2 == 1

3. 结果形状：
   C = [a1, max(a2, b1), max(a3, b2)]
```

**广播的约束**：

```
对于每个对齐的维度 i：
  constraint(dim_A[i], dim_B[i])
  
其中 constraint(x, y) = (x == y) OR (x == 1) OR (y == 1)
```

### 3.2 随机生成 vs 约束求解

**场景**：生成 `add(A, B)` 节点

**随机生成**：
```kotlin
val A = randomShape()  // [42, 57, 88]
val B = randomShape()  // [98, 122]

// 对齐：
// A = [42, 57, 88]
// B = [  , 98, 122]

// 检查：
// dim 1: 57 vs 98 → 都不是 1 且不相等 → 失败！
// dim 2: 88 vs 122 → 都不是 1 且不相等 → 失败！

// 成功率 ≈ 0.01%（维度值随机，几乎不会满足广播条件）
```

**约束求解**：
```kotlin
// 方法1：强制广播（贪婪）
// 策略：生成 A 后，调整 B 使其可广播
val A = [42, 57, 88]
val B = generateBroadcastableWith(A)  // [1, 1, 88] 或 [1, 57, 1] 或 [1, 1, 1]

// 方法2：符号约束
// 变量：A.dims[0], A.dims[1], A.dims[2], B.dims[0], B.dims[1]
// 约束：
//   A.dims[1] == B.dims[0] OR A.dims[1] == 1 OR B.dims[0] == 1
//   A.dims[2] == B.dims[1] OR A.dims[2] == 1 OR B.dims[1] == 1
// 
// 求解示例：
//   设 A.dims[1] = 57, A.dims[2] = 88
//   选择：B.dims[0] = 1, B.dims[1] = 88
//   结果：B = [1, 88] ✓
```

### 3.3 广播的"软约束"特性

**广播的特殊性**：约束是"或"关系，有多个解

```
constraint: (a == b) OR (a == 1) OR (b == 1)

给定 a = 5，合法的 b 值：
  - b = 5  （匹配）
  - b = 1  （广播）
  
给定 a = 1，合法的 b 值：
  - b = 任意值（因为 a == 1）
```

**这意味着**：
1. 约束求解不是唯一解，而是有多个解空间
2. 可以选择"最有趣"的解（测试编译器的各种场景）
3. 例如：优先选择 `b = a`（非广播）或 `b = 1`（广播）来覆盖不同路径

## 4. 矩阵乘法：为什么是硬约束？

### 4.1 矩阵乘法的形状规则

**基本矩阵乘法**（2D）：
```
A: [M, K]
B: [K, N]
C = matmul(A, B): [M, N]

约束：A.shape[1] == B.shape[0]
```

**批次矩阵乘法**（> 2D）：
```
A: [batch_dims..., M, K]
B: [batch_dims..., K, N]
C = matmul(A, B): [batch_dims..., M, N]

约束：
  1. A.shape[-1] == B.shape[-2]  (K 维匹配)
  2. batch_dims 可广播
```

### 4.2 硬约束 vs 软约束

**硬约束**：必须满足，否则操作无定义

```python
# 硬约束案例
A = [[1, 2], [3, 4]]    # shape [2, 2]
B = [[5, 6, 7]]         # shape [1, 3]

# matmul(A, B) 
# 要求：A.shape[1] == B.shape[0]
# 实际：2 ≠ 1
# 结果：错误！无法修复！
```

**软约束**：有多种满足方式

```python
# 软约束案例（广播）
A = [[1, 2, 3]]         # shape [1, 3]
B = [[4, 5, 6],
     [7, 8, 9]]         # shape [2, 3]

# add(A, B)
# 广播规则：dim 0 (1 vs 2) → OK, dim 1 (3 vs 3) → OK
# 结果：A 广播到 [2, 3]
# C = [[1+4, 2+5, 3+6], [1+7, 2+8, 3+9]]  ✓
```

**为什么矩阵乘法必须用约束求解**：

```
因为硬约束要求维度精确匹配：
  A.dims[-1] == B.dims[-2]

这不是"可以修复"的问题，而是"必须正确生成"的问题。

随机生成两个形状，恰好满足这个约束的概率 ≈ 1/128（假设维度值范围 1-128）
如果图中有多个 matmul，成功率呈指数下降。
```

## 5. 多算子组合：约束传播

### 5.1 约束传播示例

**复杂图**：

```
输入: [N, C, H, W]
  ↓
Conv2d: [N, C_out, H_out, W_out]
  ↓
Reshape: [N, C_out * H_out * W_out]  ← 约束：reshape 后的总元素数不变
  ↓
Matmul: [N, D]                        ← 约束：W2.shape[0] == C_out * H_out * W_out
  ↓
输出
```

**变量**：
- `N, C, H, W`（输入）
- `C_out, H_out, W_out`（Conv2d 输出）
- `D`（最终输出维度）

**约束链**：
```
1. Conv2d 输出形状：[N, C_out, H_out, W_out]
2. Reshape 约束：N * C_out * H_out * W_out == N * (C_out * H_out * W_out)
3. Matmul 约束：W2.shape[0] == C_out * H_out * W_out
```

**如果没有约束求解**：
- 随机生成 `W2.shape = [K, D]`，`K` 几乎不可能等于 `C_out * H_out * W_out`
- 成功率 ≈ 0

**约束求解**：
1. 选择 `N=32, C=3, H=224, W=224`
2. 选择 `C_out=64, kernel_size=3` → `H_out=222, W_out=222`
3. 推导：`W2.shape[0] = 64 * 222 * 222 = 3151296`
4. 选择 `D=10`
5. 结果：
   - 输入：`[32, 3, 224, 224]`
   - Conv2d 输出：`[32, 64, 222, 222]`
   - Reshape 输出：`[32, 3151296]`
   - Matmul 权重：`[3151296, 10]`
   - 输出：`[32, 10]`
6. **成功率 = 100%**

### 5.2 约束传播的必要性

**为什么需要约束传播**：

```
错误做法：每个节点独立生成
  节点1: matmul(A, B)   → 生成 A=[2,3], B=[3,4]
  节点2: relu(matmul_out) → 输入形状 [2,4]
  节点3: add(relu_out, C) → 随机生成 C=[5,6] ← 与 relu_out 不匹配！

正确做法：约束传播
  节点1: matmul(A, B)   → 约束：B.dims[0] == A.dims[1]
  节点2: relu(matmul_out) → 形状不变：[A.dims[0], B.dims[1]]
  节点3: add(relu_out, C) → 约束：C 可广播到 relu_out
  
  求解：
    1. 选 A.dims = [2, 3]
    2. 推导 B.dims[0] = 3, 选 B.dims[1] = 4 → B = [3, 4]
    3. 推导 relu_out = [2, 4]
    4. 生成 C 可广播到 [2, 4] → C = [1, 4] 或 [2, 1] 或 [1, 1]
```

## 6. 符号形状：为什么比贪婪实例化更好？

### 6.1 贪婪实例化的局限

**贪婪策略**：按拓扑序，每个节点立即确定形状

```
节点1: 输入 X → 选择 [32, 64]
节点2: matmul(X, W) → 约束 W.shape[0] = 64，选择 W = [64, 128]
节点3: relu(matmul_out) → 形状 [32, 128]
节点4: add(relu_out, Y) → 约束 Y 可广播到 [32, 128]，选择 Y = [1, 128]

问题：如果在节点2时选择了 W = [64, 128]，但后面发现：
  - 节点5: concat(?, W)  ← 要求 W.shape[0] == 64（已满足）
  - 节点6: matmul(W, Z) ← 约束 Z.shape[0] = 128
  - 节点7: add(Z, Q) ← Q 可广播到 [32, 128]
  
贪婪的局限：早期决策限制了后续选择
  - W.shape 已经固定为 [64, 128]
  - 后续所有依赖 W 的节点都必须适应这个形状
  - 如果后续有"反向约束"（如"希望 W.shape[1] 是偶数"），无法调整
```

### 6.2 符号形状的优势

**符号策略**：所有形状先用符号表示，最后一次性求解

```
节点1: 输入 X → 形状 [N, D]
节点2: matmul(X, W) → 形状 [N, H]，约束 W = [D, H]
节点3: relu(matmul_out) → 形状 [N, H]
节点4: add(relu_out, Y) → 形状 [N, H]，约束 Y 可广播

收集所有变量：N, D, H
收集所有约束：
  - N, D, H > 0
  - Y.dims 可广播到 [N, H]

求解：
  选择 N=32, D=64, H=128
  推导：
    X = [32, 64]
    W = [64, 128]
    relu_out = [32, 128]
    Y = [1, 128]
```

**优势**：

| 特性 | 贪婪实例化 | 符号约束求解 |
|------|-----------|-------------|
| 支持反向约束 | ❌ | ✅ |
| 全局最优 | ❌（局部最优） | ✅（全局最优） |
| 支持复杂约束 | 部分 | ✅ |
| 实现复杂度 | 低 | 中 |
| 性能 | 快 | 稍慢 |

**什么时候需要符号形状**：

1. **复杂约束**：
   - `divisible_by(shape[-1], 16)`（GPU 优化要求）
   - `shape[0] * shape[1] < 10000`（内存限制）

2. **反向约束**：
   - 图的输出形状要求
   - 用户指定的约束

3. **多解选择**：
   - 约束有多个解，选择"最有趣"的
   - 例如：选择边界值（1, 128）而非中间值

## 7. 实际案例分析

### 7.1 当前失败案例

**案例1：二元运算形状不匹配**

```python
# 从 VIOLATIONS.md
v_0: [42, 57, 88]  # 3D
v_3: [98, 122]     # 2D (mean 后降维)

subtract(v_0, v_3)
# 对齐：
#   [42, 57, 88]
#   [  , 98, 122]
# 检查：
#   dim 1: 57 vs 98 → 不匹配
#   dim 2: 88 vs 122 → 不匹配
# 结果：失败
```

**根本原因**：
- `v_3` 是 `mean` 的输出，形状由 `keepdims=False` 决定
- 生成器不知道 `v_3` 的形状会降维
- 没有检查 `v_0` 和 `v_3` 是否可广播

**约束求解方案**：

```
变量：
  v_0.dims = [D0, D1, D2]
  v_3.dims = [D3, D4]

约束：
  mean 的输入形状（假设）：[D0, D1, D2]
  mean(axis=-1, keepdims=False) 输出形状：[D0, D1]
  所以 D3 = D0, D4 = D1

  subtract 广播约束：
    dim 1: D1 vs D3 → D1 == D3 OR D1 == 1 OR D3 == 1
    dim 2: D2 vs D4 → D2 == D4 OR D2 == 1 OR D4 == 1

求解：
  选择 D0=42, D1=57, D2=88
  推导 D3=42, D4=57
  
  检查 subtract 约束：
    dim 1: 57 vs 42 → 不满足（都不是 1 且不相等）
    dim 2: 88 vs 57 → 不满足
  
  调整：
    选择 D1=1 或 D3=1 → v_0=[42, 1, 88], v_3=[42, 57]
    或选择 D2=1 或 D4=1 → v_0=[42, 57, 1], v_3=[42, 57]
```

**案例2：0输入算子固定形状**

```python
v_6 = ones([3])  # 固定形状 [3]
v_0: [89, 6]    # 2D

multiply(v_0, v_6)
# 对齐：
#   [89, 6]
#   [  ,  3]
# 检查：
#   dim -1: 6 vs 3 → 不匹配
# 结果：失败
```

**根本原因**：
- `ones([3])` 生成了固定形状
- 生成器不知道这个形状会与 `v_0` 结合
- 没有检查广播兼容性

**约束求解方案**：

```
约束：
  ones 输出形状：固定 [3]
  v_0 形状：[D0, D1]
  multiply 广播约束：D1 == 3 OR D1 == 1 OR 3 == 1

求解：
  选择 D0=89
  选择 D1=3（满足约束）
  结果：v_0 = [89, 3], ones_out = [3]
  multiply 输出：[89, 3] ✓
  
或者：
  选择 D1=1
  结果：v_0 = [89, 1], ones_out = [3]
  multiply 输出：[89, 3] ✓
```

## 8. 总结：为什么约束求解是必要的

### 8.1 三个核心原因

**1. 形状是相互依赖的变量**

```
不是：独立生成每个形状
而是：生成一组满足约束的形状
```

**2. 事后修复不可行**

```
不是：生成 → 检查 → 修复
而是：生成时就满足约束
```

**3. 复杂约束需要全局视角**

```
不是：每个节点局部决策
而是：全局收集约束 → 一次性求解
```

### 8.2 类比：数独问题

**模糊测试形状生成 ≈ 解数独**

```
数独：
  - 网格中的数字满足特定约束
  - 每行、每列、每个宫格的数字唯一
  - 不能随机填数字，必须满足约束

形状生成：
  - 图中的形状满足特定约束
  - 每个算子的输入形状兼容
  - 不能随机生成形状，必须满足约束

约束求解：
  - 数独求解器
  - 形状约束求解器
  - 本质相同：找到满足约束的解
```

### 8.3 最终答案

**问：为什么会用到约束求解？**

**答**：

1. **本质原因**：形状不是独立随机变量，而是满足特定约束的联合随机变量

2. **技术原因**：
   - 矩阵乘法等硬约束要求维度精确匹配
   - 广播等软约束有多个解，需要选择
   - 多算子组合导致约束传播

3. **工程原因**：
   - 随机生成成功率 < 50%
   - 事后修复在复杂图中不可行
   - 约束求解保证 100% 成功率

4. **类比**：
   - 就像解数独，必须满足约束
   - 就像类型检查，必须类型一致
   - 就像构建系统，必须依赖正确

**结论**：约束求解不是"可选的优化"，而是**解决形状一致性问题唯一可行的方案**。
