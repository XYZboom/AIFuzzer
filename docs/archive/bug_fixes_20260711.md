# aiFuzzer Bug 修复报告 (2026-07-11)

## 修复的问题

### Bug 1: 归约操作对 1D tensor 产生标量（0-D tensor）

**文件**: `ShapeInferer.kt`

**问题描述**:
- 归约操作（REDUCE_MAX, REDUCE_MIN, REDUCE_SUM, REDUCE_MEAN）对 1D tensor 执行 `keepdims=False` 时
- 形状推导器错误地返回 `[1]` 而不是标量 `[]`
- TVM 实际行为：对 1D tensor 归约后返回标量（0-D tensor）

**修复**:
- 移除了"避免 0-D tensor"的防御性代码
- 现在正确返回空维度列表（标量）

---

### Bug 2: GATHER 操作对 1D tensor 取单个元素产生标量

**文件**: `ShapeInferer.kt`

**问题描述**:
- `relax.op.take(tensor_1d, scalar_index, axis=0)` 应该返回标量 `[]`
- 形状推导器错误地返回 `[1]`
- TVM 实际行为：在 1D tensor 上取单个元素返回标量

**修复**:
- 移除了"确保至少 1D"的逻辑
- 现在正确返回空维度列表（标量）

---

### Bug 3: EXPAND_DIMS 生成器缺少 axis 属性

**文件**: `ShapeAdapter.kt`

**问题描述**:
- `insertExpandDims` 方法试图一次插入多个维度，但没有设置 `axis` 属性
- TVM 的 `expand_dims` 一次只能插入一个维度
- 导致形状断言失败：期望 `[1, 1]` 但实际得到 `[1]`

**修复**:
- 添加了新方法 `insertSingleExpandDims`，只插入一个维度并正确设置 `axis` 属性
- 修改 `generateWrapperSequence`，为每个要插入的维度创建单独的 EXPAND_DIMS 节点
- 使用 `buildIntAttr` 构建属性

---

### Bug 4: 归约和 SOFTMAX 操作缺少 minNdim 约束

**文件**: `ShapeConstraints.kt`

**问题描述**:
- 生成器可能对标量执行归约操作，导致 `axis -1 out of range [0, 0)` 错误
- SOFTMAX 也需要至少 1D 输入

**修复**:
- 为 REDUCE_SUM, REDUCE_MEAN, REDUCE_MAX, REDUCE_MIN 添加 `minNdim = 1` 约束
- 为 SOFTMAX 添加 `minNdim = 1` 约束

---

### Bug 5: STRIDED_SLICE 对空张量切片返回错误的维度值

**文件**: `ShapeInferer.kt`

**问题描述**:
- 对形状 `[0, 34, 105, ...]` 的空张量执行 `strided_slice(axes=[0], begin=[0], end=[-1])`
- 形状推导器返回 `[-1, 34, 105, ...]`（使用 `-1` 表示未知）
- TVM 实际返回 `[0, 34, 105, ...]`（保持空张量）
- 断言失败：`expected R.shape([-1, ...]), got R.shape([0, ...])`

**根本原因**:
- 当输入维度为 0 时，`end=-1` 规范化为 `-1 + 0 = -1`
- 计算切片长度：`sliceLength = -1 - 0 = -1`
- 导致生成负数维度值

**修复**:
- 添加特殊处理：如果输入维度为 0（空张量），切片后仍为 0
- 空张量切片保持为空张量，符合语义

---

### Bug 6: GATHER 操作缺少 minNdim 约束

**文件**: `ShapeConstraints.kt`

**问题描述**:
- 生成器对标量（0-D tensor）执行 `take(axis=0)` 操作
- TVM 报错：`axis 0 is out of range. The input tensor has 0 dimensions`
- 标量没有轴，无法执行 GATHER 操作

**修复**:
- 为 GATHER 添加 `minNdim = 1` 约束

---

### Bug 7: TILE 操作缺少 minNdim 约束

**文件**: `ShapeConstraints.kt`

**问题描述**:
- 生成器对标量（0-D tensor）执行 `tile` 操作
- 翻译器硬编码 `repeats=[1]`，导致标量被扩展为 `[1]`
- TVM 的 `tile(scalar, [1])` 会将标量扩展为 1D
- 形状推导器假设 `repeats=1`，返回标量 `[]`
- 导致断言失败：期望 `[]`，实际 `[1]`

**修复**:
- 为 TILE 添加 `minNdim = 1` 约束

## 根本原因分析

所有这些 bug 都源于同一个根本原因：**防御性地避免 0-D tensor（标量）和空张量**

开发者的意图可能是：
- 避免 0-D tensor 传播到后续算子导致问题
- 确保所有张量至少是 1-D
- 避免空张量的特殊情况

但是：
- TVM Relax 正确支持 0-D tensor（标量）和空张量（维度为 0）
- 这种"防御"导致了形状推导与实际运行时行为不一致
- 形状断言功能暴露了这些不一致

## 经验教训

1. **信任编译器的设计**：TVM Relax 设计支持 0-D tensor 和空张量，不应该防御性地规避它们
2. **形状推导必须与运行时一致**：任何差异都会被形状断言捕获
3. **添加断言是发现 bug 的好方法**：形状断言帮助快速发现了这些相关的问题
4. **一次修改一个地方**：如果有多个地方有相同的"防御性"逻辑，需要全部修复
5. **处理边界情况**：空张量（维度为 0）和标量是合法的张量，需要正确处理
