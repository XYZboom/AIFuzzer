### 三、设计决策审查

#### 3.1 两阶段架构的语义约束问题

评审之前指出：**两阶段架构（LogicGenerator + ShapeAdapter）本身不足以保证生成的程序语义合法**。LogicGenerator 只管拓扑不关心算子约束，ShapeAdapter 只管形状填充。

**本次的修复策略**: **放宽约束，允许自动适配**

| 算子 | 原约束 | 新行为 |
|------|--------|--------|
| MATMUL | ndim >= 2 | 自动扩展到 2-D |
| TRIL/TRIU | ndim >= 2 | 自动扩展到 2-D |
| GATHER | 2 inputs | 当作单输入（indices 是常量） |
| 二元算子 | 2 inputs | 当输入不足时，自己和自己运算 |

**评价**: 这是一个**务实的设计决策**。虽然生成的程序可能在语义上"不那么严格"，但：
1. 可以用于测试编译器的错误处理能力
2. 避免了两阶段架构的根本性重构
3. 通过 `maxAttempts=5` 的重试机制提高生成成功率

#### 3.2 ShapeAdapter 的重试机制

```kotlin
while (attempt < maxAttempts) {
    try {
        val shapeMap = ShapeInferer.inferGraphShapes(graph)
        // 应用形状...
        return
    } catch (e: ShapeInferenceError) {
        attempt++
        // 增加输入维度后重试
        for (input in graph.inputs) {
            val currentNdim = input.type.shape.dims.size
            if (currentNdim < 4) {
                input.type.shape = generateRandomShape(rand, minNdim = currentNdim + 1, maxNdim = 4)
            }
        }
    }
}
```

**评价**: 合理的兜底策略，避免了因形状不兼容导致整个生成失败。

#### 3.3 ShapeInferer 的放宽约束

`inferMatmulShape` 现在自动将 1-D 输入视为 2-D：

**评价**: 这使得 MATMUL 在形状推导阶段不会失败，与翻译器的行为一致（翻译器也不需要 full 替换了）。

### 四、生成样例审查

实际生成的样例：

```
=== Graph: graph_0 ===
3 nodes, 1 inputs, 2 outputs

Input v_0: [116]

REDUCE_SUM: 1 outputs -> v_1[1]
TILE: 1 outputs -> v_2[116]
GATHER: 1 outputs -> v_3[116]
```

**生成的 Python 代码**:
```python
v_0_var = relax.Var("v_0", relax.TensorType(relax.ShapeExpr([116]), "float32"))
with bb.function("graph_0", [v_0_var]):
    v_1 = bb.emit(relax.op.sum(v_0_var, axis=[-1], keepdims=false))
    v_2 = bb.emit(relax.op.tile(v_0_var, [1]))
    v_3 = bb.emit(relax.op.take(v_0_var, relax.const(0, dtype="int64"), axis=0))
    bb.emit_func_output((v_2, v_3))
```

**观察**:
1. ✅ 形状正确填充：`v_0: [116]`，`v_1: [1]`（reduce 后）
2. ✅ Python 代码语法正确
3. ⚠️ `keepdims=false` 应该是 `keepdims=False`（Python 布尔值）
4. ⚠️ `axis=[-1]` 是负索引，TVM 支持
